大模型领域，造概念的速度和模型发版的速度一样快。

2026年6月初,Anthropic 的 Boris Cherny(Claude Code 的创建者)说了一句话,被广泛引用:"我现在不再亲自给 Claude 写提示了,而是有一些循环在运行,它们才是真正在给 Claude 下指令、决定下一步做什么的角色。" Google Chrome 的工程负责人 Addy Osmani 把这句话和另一位开发者 Peter Steinberger(PSPDFKit/Nutrient 创始人)的类似观点结合起来,提炼成了"Loop Engineering"这个概念,并迅速在博客、Substack 和一个 GitHub 参考仓库里传播开来,短时间内成了社区内的标准用语。Tosea.ai 的报道提到最初引发讨论的那篇文章在几天内浏览量超过650万,扩散速度很快。

循环工程（Loop Engineering）——用你自己设计系统来提示智能体，取代你亲自去提示它。 这里的"循环"可以理解为一个递归目标：你定义一个目的，AI 不断迭代直到完成。我相信这可能是我们与编码智能体协作的未来。不过，它仍然处于早期阶段，我持保留态度，而且你绝对必须小心 token 成本（不同使用模式的 token 消耗差异巨大，取决于你是 token 富裕还是匮乏），所以我想拆解一下它是什么以及它意味着什么。

Peter Steinberger 最近说："你不应该再亲自提示编码智能体了。你应该设计循环来提示你的智能体。"同样地，Anthropic 的 Claude Code 负责人 Boris Cherny 也说："我不再提示 Claude 了。我有循环在运行，它们提示 Claude 并决定下一步做什么。我的工作是写循环。"

好吧，这到底是什么意思？

在过去大约两年里，你从编码智能体那里得到结果的方式是：写一个好的提示词，提供足够的上下文。你输入一段话，阅读返回的内容，再输入下一段。智能体是一个工具，你全程握着它，一个回合接一个回合。那个时代已经结束了——至少有些人认为它即将结束。

现在，你构建一个小型系统：它找到工作、分配任务、检查结果、记录已完成事项，然后决定下一步做什么。你让这个系统去驱动智能体，而不是你自己去驱动。我之前写过它的近亲——智能体套件工程（Agent Harness Engineering），即打造单个智能体运行的环境；以及工厂模型（Factory Model），即构建软件的系统。循环工程位于套件之上的一层：套件加上定时器，它会孵化小助手，并且能自我喂养。

让我惊讶的是，这已经不再是一个工具层面的问题了。一年前，如果你想要一个循环，你得写一堆 bash 脚本，然后永远维护那堆脚本——它属于你，也只属于你。现在，这些组件直接内置于产品中。Steinberger 列出的清单几乎完美映射到 Codex 应用，然后几乎同样映射到 Claude Code。一旦你注意到结构是相同的，你就不会再争论用哪个工具了——你只需设计一个循环，无论你坐在哪个工具里它都能工作。

五个组件，以及一些说明
一个循环需要五样东西，外加一个记录状态的地方。先列出清单，然后做映射。

自动化（Automations）
按计划自动运行，自行完成发现和分类。
工作树（Worktrees）
让两个并行工作的智能体不会互相踩脚。
技能（Skills）
记录项目知识，智能体不用再靠猜测。
插件和连接器（Plugins & Connectors）
将智能体接入你已经在使用的工具。
子智能体（Sub-agents）
一个负责出主意，另一个负责检查。
然后是第六样东西——记忆（Memory）。一个 Markdown 文件，或者一个 Linear 看板——任何存在于单次对话之外、记录已完成和待办事项的东西。听起来简单得不像回事，但这是每个长时间运行的智能体都依赖的技巧。我在长时间运行智能体中深入探讨过：模型在每次运行之间会忘记一切，所以记忆必须存在磁盘上，而不是在上下文中。智能体会忘记，但仓库不会。

两款产品现在都具备了这五个组件。

自动化——这是心跳
自动化才是让循环成为真正循环的东西，而不仅仅是你做过的一次运行。在 Codex 应用中，你在 Automations 标签页创建一个自动化，选择项目、要运行的提示词、运行频率，以及是在本地检出上运行还是在后台工作树上运行。有发现的运行结果进入 Triage 收件箱，没有发现的运行结果会自动归档——这很贴心。OpenAI 内部用它们来做无聊的事情：每日问题分类、汇总 CI 失败、编写提交简报、查找上周有人添加的 bug。而且自动化可以调用技能，这样你可以保持重复性工作的可维护性——你执行 $skill-name，而不是把一堵巨大的指令墙粘贴到永远不会有人更新的计划里。

Claude Code 通过调度和 hooks 达到了同样的效果。你可以用 /loop 按间隔运行提示词或命令，可以安排 cron 任务，可以在智能体生命周期的特定点通过 hooks 触发 shell 命令，或者如果你希望在你合上笔记本后它继续运行，可以把它推到 GitHub Actions。完全相同的思路：你定义一个自主任务，给它一个节奏，结果会送到你面前——你不再需要四处检查。

还有一个值得了解的会话内原语，它更接近这篇文章的核心。/loop 按节奏重复运行。/goal 持续运行直到你写的条件真正满足，每轮之后有一个独立的小模型检查是否完成——这样写代码的智能体就不会给自己打分。你给它类似"test/auth 中的所有测试通过且 lint 干净"这样的条件，然后走开。Codex 也有同样的功能，也叫 /goal，它会跨轮次持续工作直到一个可验证的停止条件成立，支持暂停、恢复和清除。相同的原语，两个工具都有——这几乎成了整篇文章的模式。

所以，这部分负责发现工作。循环的其余部分负责对发现采取行动。

工作树——让并行不会变成混乱
一旦你运行超过一个智能体，文件就开始冲突——这就是失败的原因。两个智能体写入同一个文件，和两个工程师在没沟通的情况下提交到同一行代码一样令人头疼。Git 工作树解决了这个问题：它是一个独立的工作目录，拥有自己的分支，共享同一个仓库历史——所以一个智能体的编辑物理上无法触及另一个智能体的检出。

Codex 内置了工作树支持，多个线程可以同时访问同一个仓库而不会互相碰撞。Claude Code 通过 git worktree、--worktree 标志（在自己的检出中打开会话），以及在子智能体上设置 isolation: worktree 来实现同样的隔离——这样每个助手都能获得一个全新的检出，用完自动清理。我在编排税（The Orchestration Tax）中写过这方面的人性面：工作树消除了机械碰撞，但你仍然是天花板——你的审查带宽决定了你能实际运行多少个，而不是工具。

技能——让你不再每次都要解释项目
技能是你不再像金鱼一样每次会话都要重新解释项目上下文的方式。两个工具使用相同的格式：一个包含 SKILL.md（内含指令和元数据）的文件夹，以及可选的脚本、参考资料和资源。Codex 在你用 $ 或 /skills 调用时运行技能，或者当你的任务匹配技能描述时自动运行——这就是为什么一个紧凑、朴实的描述胜过花哨的描述。Claude Code 以同样的方式运行，我在智能体技能中写过这个模式。

技能也是让你的意图不再反复消耗成本的地方。我在意图债务（The Intent Debt）中论证过：智能体每次会话都是从零开始，它会用自信的猜测填补你意图中的任何漏洞。技能就是写在外部的那份意图——约定、构建步骤、"我们不用这种方式因为那次事故"——写一次，智能体每次运行都会读到。没有技能，循环每次都会从零重新推导你的整个项目；有了技能，它会不断积累。

有一点要搞清楚：技能是创作格式，插件是分发方式。当你想要跨仓库共享技能或打包几个技能时，你把它们打包成插件。Codex 如此，Claude Code 也是如此。

插件和连接器——循环触及你的真实工具
一个只能看到文件系统的循环是一个微小的循环。连接器（基于 MCP 构建）让智能体读取你的问题追踪器、查询数据库、访问 staging API、在 Slack 中发送消息。Codex 和 Claude Code 都支持 MCP，所以你为一个工具写的连接器通常也能在另一个工具中工作。而插件将连接器和技能打包在一起，这样你的队友一次安装就能获得你的整个配置，而不必凭记忆重建一切。

这就是"智能体说'这是修复方案'"和"循环自动打开 PR、关联 Linear 工单、并在 CI 变绿后自动在频道中通知"之间的区别。连接器是循环能在你的实际环境中行动（而不仅仅是告诉你它想做什么）的原因。

子智能体——让制造者和检查者分开
循环中最有用的结构性组件，毫无疑问，是把写代码的人和检查代码的人分开。写代码的模型给自己打分时实在太客气了。第二个拥有不同指令（有时是不同模型）的智能体能抓住第一个智能体自欺欺人的问题。

Codex 只在你要求时才会生成子智能体，同时运行它们，然后将结果合并回一个答案。你在 .codex/agents/ 中将自己的智能体定义为 TOML 文件，每个都有名称、描述、指令以及可选的模型和推理努力度——所以你的安全审查员可以用一个高努力度的强模型，而你的探索者可以是一个快速的只读智能体。Claude Code 在 .claude/agents/ 中使用子智能体和在它们之间传递工作的智能体团队做同样的事情。两者中常见的分工是：一个智能体探索，一个实现，一个对照规范验证。

我已经两次提出这个观点：一次是编码智能体管弦乐队（The Code Agent Orchestra），一次是对抗性代码审查（Adversarial Code Review）。它在循环中特别重要的原因是：循环在你没看着的时候运行，所以一个你真正信任的验证者是你敢走开的唯一理由。子智能体确实会消耗更多 token——因为每个都有自己的模型和工具工作——所以只在第二意见值得付费的地方使用它们。这基本上也是 Claude Code 的 /goal 在底层做的事情：一个全新的模型决定循环是否完成，而不是做工作的那个——制造者和检查者的分离被应用到了停止条件本身。

一个循环长什么样
把各部分拼在一起，一个线程就变成了一个小控制面板。这是我一直在使用的一个形态。

一个自动化每天早上在仓库上运行。它的提示词调用一个分类技能，读取昨天的 CI 失败、未解决的问题、最近的提交，并将发现写入一个 Markdown 文件或 Linear 看板。对于每个值得做的发现，线程打开一个隔离的工作树，派一个子智能体起草修复方案，第二个子智能体对照项目技能和现有测试审查那份草稿。
连接器让循环自动打开 PR 并更新工单。循环无法处理的任何内容都会落入我的 Triage 收件箱。状态文件是整个循环的脊柱——它记录了尝试过什么、通过了什么、还有什么未完成——这样第二天早上运行时，可以从今天停下的地方继续。

看看你实际做了什么：你设计了一次。你没有提示任何这些步骤。这就是 Steinberger 的整个观点变成了现实——而且它在 Codex 和 Claude Code 中是同一个循环，因为组件是相同的组件。

循环仍然不能为你做什么
循环改变了工作，但它并没有把你从工作中删除。而且有三个问题实际上会随着循环变得更好而变得更加尖锐，而不是更轻松。

验证仍然在你身上。一个无人值守运行的循环，也是一个无人值守犯错的循环。你把验证者子智能体从制造者中分离出来的全部原因，就是让循环的"完成了"有点意义——即便如此，"完成了"也只是一个声明，而不是一个证明。我一直在重复AI 时代的代码审查中的同一句话：你的工作是交付你确认能工作的代码。

如果你放任不管，你的理解力仍然会退化。循环交付你没有写的代码越快，已存在的代码和你实际理解的东西之间的差距就越大。这就是理解债务（Comprehension Debt）——一个顺畅的循环只会让它增长得更快，除非你阅读循环生成的东西。

舒适的姿势是最危险的。当循环自己运行时，你很容易不再有自己的判断，只是接受它返回的任何东西。我称之为认知投降（Cognitive Surrender）。设计循环——当你带着判断力去做时——是解药；当你为了逃避思考而去做时——是催化剂。同样的动作，相反的结果。

构建循环。保持工程师的身份。
我认为这是我们工作方式演变的一个预览。话虽如此，如果我不亲自审查代码，或者如果我完全依赖自动化循环来修复问题，我的产品质量会受损。我可能会陷入一个向下的螺旋，不断把自己挖进更深的坑里。

尽管如此，去设置你的循环吧，但不要忘记直接提示你的智能体也同样有效。关键在于找到正确的平衡。

循环也可能因为使用者的不同而产生截然不同的结果。两个人可以构建完全相同的循环，却得到完全相反的结果。一个人用它来加速理解深入的工作，另一个人用它来逃避理解工作本身。循环不知道其中的区别。你知道。

这就是为什么循环设计比提示工程更难，而不是更容易。Cherny 的观点不是工作变简单了，而是杠杆点移动了。

构建循环。但要像一个打算保持工程师身份的人那样去构建它——而不仅仅是那个按下"开始"按钮的人。
